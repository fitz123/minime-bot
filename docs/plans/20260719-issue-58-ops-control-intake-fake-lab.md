# ADR-099 Ops Final Feature: Telegram Control, Alertmanager Intake, Availability Checks, and Fake Fault Lab

## Overview

Implement the final ADR-099 feature PR on fresh `main` after PR #102. This phase adds the dedicated second-token ops Telegram control plane with durable safe-boundary steering, broad authenticated Alertmanager intake, package-owned goal-appropriate availability done-check registrations on the PR #102 composite verifier, and the complete batched fake fault lab that runs the full orchestration-regression matrix as one deterministic aggregate result.

This is ONE coherent feature PR. It does not release, deploy, perform private rollout, start or activate any production listener, run a production outage drill, use sudo, create or mutate credentials/tokens, or perform legacy cleanup. The ops worker remains inactive by default: nothing outside an explicit `worker start` invocation with trusted embedding dependencies runs a worker, poller, or listener. Public code may add validated config/schema/secret-reference surfaces and examples with placeholders only — never real identifiers, secrets, private paths, destinations, or PII.

Primary issue: https://github.com/fitz123/minime-bot/issues/58
Base source: `7c53a7a` (`origin/main` when this plan was prepared)

## Development Approach

- TDD for every schema, ledger, steering, control, intake, verifier, and lab behavior.
- Keep trusted package code — never task payloads, Telegram text, or alert payloads — in control of templates, profiles, verifiers, done-check components, probe targets, and token sources. External payloads and operator text are evidence, not authority (operator steering is trusted evidence but still not executable configuration).
- Preserve every PR #101/#102 invariant: atomic v-schema snapshots, exclusive whole-cycle custody, fixed lifecycle slots/receipts, continuous authorization, quota admission/reset waits, context/capability parity, and fresh aggregate composite PASS as the only route to DONE.
- Preserve the second-token/no-shared-getUpdates invariant structurally: `src/ops-worker/**` must not import `grammy`, `src/telegram-bot.ts`, or `src/telegram-adapter.ts`; the ops poller uses only its own validated token source and injected fetch; the primary bot code is not modified.
- Durable-before-ack: an ops Telegram update is persisted (and its steering/task effect recorded) before its update id is covered by the next `getUpdates` offset and before any reply is sent. Crash between persist and ack must make Telegram redelivery an idempotent no-op.
- Normal steering is consumed at the next safe boundary; it is never written into a running Pi/Ralphex child's stdin or session. Only the explicit cancel/interrupt path stops a proven owned process group.
- Keep one globally serial custody owner. Preserve normalized immutable repository/resource identity; add no lanes, workflow engine, broker, SQLite, leases, second supervisor product, root helper, generic shell done-check, or task-manager scanner (#70 stays deferred). `operator-telegram` task creation stays dormant.
- No fallback/legacy dual path; native Alertmanager-to-Telegram delivery stays independent and unmodified.

## Technical Boundaries

### Telegram control

- Dedicated second BotFather token from the ops control config only; the primary `telegramToken*` sources are never read and the primary grammY poller is never shared or modified. `src/ops-worker/**` must not import grammY or primary Telegram modules.
- Persist before acknowledging: ledger + steering/task effects are durable before offset advance or reply. Redelivery and restart are idempotent by update id and steering id.
- Steering is consumed at the next safe boundary through the attempt prompt; nothing is injected into a running child. Cancel/interrupt stops only a proven owned process group; ambiguity keeps the global fence.
- Telegram text can steer only existing tasks; it cannot create tasks or configure commands, destinations, URLs, models, tools, or registered contracts. Ordinary operator prose and references remain bounded data. `operator-telegram` task submission stays dormant for the future #70 adapter.

### Alertmanager intake

- Broad host/service surface with one registered template/profile/check set — no per-alert remediation profiles, no static runbook/action allowlist, no second policy engine.
- Payloads are strictly bounded, validated, deduplicated, and stored as untrusted evidence; trust comes only from the configured source identity and bearer authentication. Loopback bind only.
- The native Python Alertmanager→Telegram path stays independent and unmodified.

### Verification

- Only the PR #102 typed composite registry executes checks; fresh stable evidence (monitoring freshness + resolved alerts + direct stable health over a trusted window) decides PASS. Typed product/not-ready/defer/verifier/query/timeout distinctions are preserved end to end.

### Inactivity and scope

- No release, deploy, private rollout, production listener activation, outage drill, sudo, credential creation/mutation, or legacy cleanup. `main.ts` and the primary bot runtime are untouched. `worker start` still fails closed without trusted embedding dependencies; all new listeners/pollers exist only inside an explicitly started worker with explicit control config.
- Global gates unchanged: literal sudo, irreversible/destructive cleanup, credential/access-control mutation, financial, and newly expanded public/external scope remain separately gated. No new operator acceptance criteria are invented.

## Success Criteria

- v1–v4 snapshots migrate purely to v5; steering/pause/interrupt state is durable, bounded, idempotent, and crash-safe.
- Ops Telegram control works end to end against fakes with durable-before-ack semantics, allowlist enforcement, safe-boundary steering, proven-group cancel/interrupt, and receipt-safe reporting — without touching the primary bot or its token.
- Authenticated intake converts valid alert groups into idempotent evidence-only task submissions and rejects everything else; native delivery is untouched.
- The availability composite check passes only on fresh stable multi-component evidence; every typed non-PASS outcome is distinguished.
- The fault lab runs the complete named matrix in one batch and yields one deterministic aggregate PASS with zero real network activity.
- The package remains inactive by default and full validation is green.
